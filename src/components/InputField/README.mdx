# Input Field

An input field is a fundamental form component used for text input. It supports different sizes, states, icons, and validation messages.

## Best Practices

- Always use a **label** to describe the input field.
- Use **placeholder text** sparingly. It should not replace labels.
- Show **error messages** inline to help users correct mistakes quickly.
- Use **icons** to enhance usability but avoid unnecessary clutter.
- Maintain consistent spacing and sizes across all input fields.

## Input Field Guidelines

### Default State

- **Purpose**: For users to input standard text.
- **Use Cases**:
  - Collecting names, email addresses, or short text-based input.
- **Best Practices**:
  - Keep placeholder text clear and concise.
  - Always pair an input field with a descriptive label for accessibility.

<Story of={InputFieldStories.Default} />



### Error State

- **Purpose**: Indicates that the input is invalid or incorrect.
- **Use Cases**:
  - Required fields left empty.
  - Invalid format (e.g., invalid email, incorrect password).
- **Best Practices**:
  - Display a clear and actionable error message.
  - Use a visually distinct border and text color (e.g., red) for errors.

<Story of={InputFieldStories.ErrorState} />


### Disabled State

- **Purpose**: Prevents user interaction.
- **Use Cases**:
  - Fields temporarily unavailable due to form logic or permissions.
- **Best Practices**:
  - Use a muted color to signify the disabled state.
  - Ensure users understand why the field is disabled.

<Story of={InputFieldStories.Disabled} />


### Input with Icons

- **Purpose**: Enhances usability with visual indicators.
- **Use Cases**:
  - **Prefix Icons**: For search, email, or password fields.
  - **Suffix Icons**: For clear actions like toggling visibility or clearing the input.
- **Best Practices**:
  - Ensure icons are aligned and maintain consistent spacing.
  - Avoid using too many icons that could clutter the field.

### Input with Hint Message

- **Purpose**: Provides inline validation feedback to guide users.
- **Use Cases**:
  - Display error or success states dynamically based on user input.
- **Best Practices**:
  - Pair validation states with clear error or hint messages.
  - Keep validation messages concise and actionable.

<Story of={InputFieldStories.WithHintMessage} />

### Sizes

- **Purpose**: Adjusts the input field dimensions based on design needs.
- **Sizes**:
  - **Small**: Compact inputs for dense forms.
  - **Medium**: Standard size for most use cases.
  - **Large**: Prominent inputs for high-importance fields.
